# ASOQA — scroll-scrub одностраничный сайт-история

**Дата:** 2026-07-07 · **Проект:** ASOQAsite · **Тип:** статический сайт (без фреймворков)

## 1. Суть

Промо-сайт компании ASOQA (разработка мобильных приложений). Одно фиксированное
окно на весь экран. 8 видеоклипов склеены в **одну непрерывную ленту кадров** и
проигрываются **исключительно скроллом** (scroll-scrub, как на страницах Apple):
позиция скролла = конкретный кадр. Скролл вниз — лента вперёд, вверх — назад, стоп —
кадр замирает. Поверх ленты на нужных участках всплывают минималистичные «стеклянные»
плашки с текстом на RU/EN (переключатель сверху). Приоритет №1 — **никаких подвисаний**.

Клипы сделаны стыкующимися (конец одного ≈ начало следующего), поэтому лента
выглядит как один непрерывный ролик, а не как отдельные сцены.

## 2. Карта ленты (8 клипов → 8 плашек)

Порядок = порядок файлов. Каждый клип занимает участок глобальной ленты
пропорционально своей длительности.

монтажные склейки, для скраба это просто переход кадр-в-кадр, читается как склейка в ролике.

## 3. Архитектура

Чистый **HTML + CSS + JS**, ноль внешних зависимостей → лёгкий, портируемый, годится
под GitHub Pages. Компоненты, каждый с одной зоной ответственности:

```
index.html
  ├─ assets/frames/sceneNN/frame_XXXX.jpg   (кадры для скраба, генерируются)
  ├─ assets/video/*.mp4                       (пережатые оригиналы: постеры/резерв)
  ├─ css/style.css
  └─ js/
       ├─ config.js       — манифест сцен: путь, число кадров, диапазон прогресса, копирайт
       ├─ i18n.js         — словарь RU/EN, переключатель, запоминание выбора (localStorage)
       ├─ loader.js       — прелоадер + прогресс + последовательный префетч кадров
       ├─ scrubber.js     — движок скраба: scroll→глобальный кадр→drawImage на canvas
       ├─ panels.js       — показ/скрытие плашек по глобальному прогрессу
       └─ main.js         — сборка: инициализация, порядок запуска
```

### Модель скролла
- `<body>` имеет высоту `N * 100vh` (N — «длина» ленты в экранах, задаёт скорость
  скраба; напр. ~800vh на всю ленту — подберём при верстке для комфортного темпа).
- Сцена (canvas + плашки + топ-бар) — `position: fixed`, всегда на весь экран.
- Глобальный прогресс `p = scrollY / (scrollHeight - innerHeight)` ∈ [0,1].
- `p` → индекс глобального кадра `f = round(p * (totalFrames-1))` → `drawImage(frame[f])`.

### Движок скраба (scrubber.js) — плавность и «не виснет»
- Скролл-события НЕ рисуют напрямую. Есть цель `targetFrame` (из скролла) и текущий
  `renderFrame`, который в цикле `requestAnimationFrame` плавно догоняет цель (lerp) —
  убирает рывки и «лестницу».
- Рисуем на один `<canvas>` через `drawImage`, `object-fit: cover` вручную (масштаб под
  вьюпорт, центрирование), чтобы кадр всегда закрывал экран без полей.
- **Память:** кадры хранятся как сжатые `Image` (JPEG в памяти ~50 МБ, НЕ распакованные;
  распаковка всех 627 кадров в bitmap заняла бы гигабайты — так нельзя). Перед отрисовкой
  делаем `img.decode()` на небольшое окно вперёд по направлению скролла (lookahead ~4–8
  кадров), чтобы декодирование не блокировало кадр анимации.
- **Граница загрузки:** скраб клампится к последнему загруженному кадру. Если пользователь
  «обогнал» префетч — держим последний готовый кадр (не мигаем, не виснем), догрузка
  подтянет остальное. Мгновенно, без блокировки.

### Прелоадер и префетч (loader.js)
- Фирменный экран ASOQA + процент. Грузит кадры **1-й сцены** → снимает экран → показывает
  подсказку «▼ листайте».
- Дальше последовательный воркер догружает сцены 2..8 **вперёд по ленте** (естественно
  совпадает с направлением скролла), держась впереди позиции пользователя.
- Стратегия по умолчанию — прогрессивная (быстрый старт). Переключаемо на «полный
  предзагруз всех кадров за прогресс-баром» (гарантированный ноль-хитчинг ценой более
  долгого старта) — константа в config.js.

## 4. Конвейер сжатия видео → кадры

Инструмент: `ffmpeg` (portable-бинарник через npm `ffmpeg-static`, уже установлен;
sudo/apt недоступны). Для каждого клипа:

1. Нормализовать fps до скраб-fps (12 по умолчанию), масштабировать до ширины 1280
   (десктоп-слой) и 854 (мобильный слой), убрать звук.
2. Выгрузить в JPEG-последовательность `frame_%04d.jpg`, качество `-q:v 4..6` (тюним по
   весу/картинке).
3. Дополнительно: пережать оригиналы в аккуратный `.mp4` (H.264, CRF ~26, `-movflags
   +faststart`, scaled 1080p) — как постер первого кадра и резерв.

Ожидаемый вес: старт ≈ 3–4 МБ (сцена 1), полная лента ≈ 25–45 МБ, размазанная по скроллу
и лениво подгружаемая. Существенно легче исходных ~100 МБ.

> Развилка веса зашита в параметры (fps, ширина, `-q:v`). Дефолт — баланс; можно сдвинуть
> в «максимально легко» без заметной потери под движением (кадр под плашкой статичен).

## 6. Визуал и адаптив

- Топ-бар: минимал, фикс, лёгкий градиент-подложка для читаемости на любом кадре.
- Плашки центрируются/позиционируются так, чтобы не перекрывать ключевой объект кадра
  (лампочку, планшет, ракету) — позиция задаётся в config.js на сцену.
- Адаптив: мобильный слой кадров (854px), плашки перекомпонуются под узкий экран,
  размеры шрифта на `clamp()`. Скролл-скраб работает и на тач (scroll driven).
- Тема плашек — светлый текст на затемнённом стекле по умолчанию (читается и на белых, и
  на тёмных сценах); проверяется на самых светлых (1,2) и самых тёмных (7) кадрах.

## 7. Гарантии «не подвисает» (сводно)

1. Скраб рисует preload-JPEG на canvas — не зависит от video-seek (главный источник
   лагов на телефонах устранён).
2. rAF + lerp сглаживают, скролл не блокирует рендер.
3. Lookahead `decode()` — декодирование не в кадре анимации.
4. Кламп к загруженной границе — при обгоне держим кадр, не мигаем.
5. Кадры в памяти сжатыми (не bitmap) — нет утечки памяти.

## 8. План проверки (verification)

Локальный статик-сервер + Playwright:
- Прелоадер показывается и уходит после загрузки 1-й сцены.
- Скролл вниз двигает кадры вперёд, вверх — назад; стоп фиксирует кадр.
- Быстрый скролл до конца — нет белых экранов/зависаний (кламп работает).
- Переключатель RU/EN меняет весь текст, выбор запоминается после перезагрузки.
- Плашки появляются/исчезают на правильных участках, не перекрывают ключевой объект.
- Мобильная ширина (напр. 390px) — верстка и скраб корректны.
- Скриншоты ключевых сцен (1, 5, 7, 8) на RU и EN — показать владельцу перед финалом.

## 9. Открытые вопросы / решаемо по ходу

- Точная длина ленты в `vh` и позиции плашек — подберутся при верстке под комфортный темп.
- Финальный контакт для CTA — заглушка до предоставления (Telegram/почта).
- Дефолт веса (fps/ширина/качество) — стартуем на балансе, тюним после первого прогона.
- Опционально: звук в финале (клип 8 со звуком) — по умолчанию выкл; можно добавить
  кнопку «включить звук» на последнем участке. Решим после базовой сборки.
